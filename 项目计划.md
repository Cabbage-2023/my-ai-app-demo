# 项目计划

## Context

当前项目已跑通基础 RAG 闭环（addResource → 存储 → getInformation → $vectorSearch → 回复），但停留在"入门级"。用户希望在 RAG 和 Tool Calling 上做出深度，以在面试中展示工程判断力。

### 新定位

不再是"从 Bangumi API 拉几条数据玩 demo"。

**新定位：** 针对 Galgame 小众领域，从 Bangumi 上的碎片化讨论/评价中构建垂直 RAG 知识库，让 AI 能基于真实的玩家评价回答角色/作品相关问题。

### 核心转变

1. 数据来源：结构化 API → 非结构化论坛帖子/评价（Bangumi）
2. 数据获取：手动复制 → 爬虫脚本（`scripts/crawl/crawler.ts`）
3. 效果验证：主观感受 → 评估脚本量化对比（`scripts/eval/evaluate.ts`）
4. 输出形式：README 只有代码 → README 包含架构图、效果数据、决策权衡

### 全部功能清单（按优先级排列）

| 优先级 | 功能 | 说明 |
|--------|------|------|
| **P0** | **爬虫 & 数据清洗** (`scripts/crawl/crawler.ts`) | 爬取 Bangumi 条目下的讨论/评价 |
| **P0** | **评估脚本** (`scripts/eval/evaluate.ts`) | 20-30 条 QA 对，跑 Hit Rate / MRR，对比各功能效果 |
| **P0** | **README 重写** | 架构图、选型辩护、效果数据表格、已知局限 |
| P1 | 文本分块（Chunking） | 中文感知递归分割器 |
| P1 | 去重写入（Dedup） | 写前查 metadata 去重 |
| P2 | 自查询检索（Self-Querying / Metadata Filter） | 按 type/tags 过滤 |
| P3 | 查询重写（Query Transformation） | 改写口语问题为检索友好形式 |
| P3 | 重排序（Reranking） | SiliconFlow rerank API，Top10→Top3 精排 |
| P4 | 混合搜索（Hybrid Search） | $vectorSearch + $text，RRF 合并 |
| P4 | 前端体验增强 | 引用溯源、Token 成本可视化、Tool Calling 深度应用 |

> 每做完一个 P1-P4 的功能，跑一次评估脚本看指标是否提升。数据比感觉重要。

- `scripts/lib/crawl/parsers/blog-parser.ts` — 长评全文解析器

### 0.1 认证

两种认证方式：

**① API 访问（游戏简介 + 角色）** — `Authorization: Bearer <token>`
```env
BANGUMI_API_TOKEN=
```
在 `.env` 中添加，调用 `api.bgm.tv` 时带上。Bearer Token 长期有效，可访问所有内容（含 R18），不受 cookie 过期影响。

**② HTML 访问（评论/文章）** — Cookie（`chii_sec_id` + `chii_sid`）
```env
BANGUMI_COOKIE_SEC_ID=
BANGUMI_COOKIE_SID=
```
作为 `.env` 兜底。但 `chii_sid` 变动极快（约 1 分钟），因此使用 **Puppeteer（无头浏览器）** 实现实时 cookie 刷新：

- 运行 `pnpm tsx scripts/crawl/cookie-refresher.ts` 打开 Edge 浏览器（复用 `puppeteer-profile` 保存的会话）
- 首次需手动输入验证码登录，之后会话自动保持
- 浏览器保持打开，每 30 秒**新建一个页面**刷新 cookie 到 `scripts/data/cookies.json`
- 每次刷新新建页面 → 获取 cookie → 关闭页面，避免 detached frame 问题
- `fetchHTML` 优先从 `cookies.json` 读取实时 cookie，无文件时退回到 `.env`
- R18 页面（评论/文章）`fetch()` 仍被 Bangumi 拦截，需通过 `browser-fetcher.ts` 走 Puppeteer CDP 连接实际浏览器获取

> **Cookie 有效期：**
> - `chii_sec_id` → **约 1 年**，几乎不用管
> - `chii_sid` → **约 1 分钟过期**，Puppeteer 浏览器自动刷新，无需手动导出

### 0.2 要爬取的内容与对应页面结构

**数据获取方式总览：**

| 数据类型 | 主要手段 | NSFW 处理 |
|---------|---------|-----------|
| 排行榜（游戏 ID 列表） | 爬虫 HTML → 无需 cookie | 无影响 |
| 游戏简介 | **API** `GET /v0/subjects/{id}`（Bearer Token） | API 完全不受限 |
| 玩家短评 | 爬虫 HTML `/subject/{id}/comments` | **Puppeteer 实时 cookie** 可访问全部评论 |
| 玩家长评 | 爬虫 HTML `/subject/{id}/reviews` | 同上 |
| 角色信息 | **API** `GET /v0/subjects/{id}/characters` + `GET /v0/characters/{id}`（Bearer Token） | API 完全不受限 |
| 角色评论 | HTML 角色页 `/character/{id}` → 无需 cookie（含 R18） | 公开可访问 |

**① 排行榜 → 获取前 300 名游戏 ID**

| 项目 | 内容 |
|------|------|
| URL | `https://bangumi.tv/game/tag/Galgame/?sort=rank&page={n}` |
| 分页 | 总共 61 页，每页约 24-30 条，前 300 名需遍历前 10 页 |
| 目标数据 | 游戏名、排名、评分、`/subject/{id}` 链接、封面图 URL |
| 解析元素 | `#browserItemList li.item` 内的 `h3 a.l`（标题）、`.rank`（排名）、`.rateInfo`（评分） |
| 翻页元素 | `.page_inner a.p`，最后一页的标记是 `›|` 链接到 `page=61` |
| **筛选条件** | **仅保留评分人数 > 500 的条目**（`.rateInfo .tip_j` 中括号内的数字，如 `(3523人评分)`），过滤掉小众作品，保证每条游戏都有足够的评论数据 |

**② 游戏简介（game_intro）**

| 项目 | 内容 |
|------|------|
| URL | `https://bangumi.tv/subject/{subject_id}` |
| 目标数据 | 游戏名称、剧情梗概、信息表（开发/发行/剧本/类型等）、标签、评分 |
| 解析关键元素 | `#subject_summary` 内文本（剧情简介）、`#infobox` 各 `li`（开发信息）、`.subject_tag_section`（标签）、`.global_score`（评分） |
| 图片 | 封面图在 `#bangumiInfo .infobox .thickbox img` 的 `src`，格式 `//lain.bgm.tv/r/400/pic/cover/...` |
| 角色链接 | 主页底部的 `.crtList` 中每个角色的 `href="/character/{id}"` |

**③ 玩家短评（吐槽 / comments）**

| 项目 | 内容 |
|------|------|
| URL | `https://bangumi.tv/subject/{subject_id}/comments?page={n}` |
| 目标数据 | 每条评论文本、评分（星星数）、发表时间 |
| 解析关键元素 | `.item.clearit` 内的 `p.comment`（文本）、`.starstop-s`（评分）、`small.grey`（时间） |
| 分页 | URL 参数 `page={n}` |

**④ 玩家长评（文章 / reviews）**

| 项目 | 内容 |
|------|------|
| URL | `https://bangumi.tv/subject/{subject_id}/reviews` |
| 目标数据 | 文章标题、正文摘要、作者、回复数 |
| 解析关键元素 | `.entry-list .item` 内的 `h2.title a`（标题）、`.content`（正文摘要） |
| 注意 | 长评内容可能被截断，如需完整内容需爬 `/blog/{blog_id}` |

**④-2 长评全文（blog）**

| 项目 | 内容 |
|------|------|
| URL | `https://bangumi.tv/blog/{blogId}` |
| 目标数据 | 长评正文纯文本 |
| 认证 | 非 R18 公开可访问；R18 需 cookie 或 Puppeteer 浏览器 fallback |
| 解析容器 | `#entry_content` |
| 清洗 | 去除 `<br>`、`&nbsp;`、表情 `<img>`，保留纯文本 |
| 脚本 | `crawl-full-reviews.ts` 遍历已存 review 列表，逐篇补爬 `fullContent` |
| 注意 | 本节后续可能需要补爬登录态的完整文章内容 |

**⑥ 角色简介（character）**

| 项目 | 内容 |
|------|------|
| 获取角色 ID | 从游戏详情页（`/subject/{id}`）底部的角色列表，或 `/subject/{id}/characters` 页面 |
| 角色列表页 URL | `https://bangumi.tv/subject/{subject_id}/characters` |
| 角色详情页 URL | `https://bangumi.tv/character/{character_id}` |
| 目标数据 | 角色名、详细描述、性别/生日等属性、CV |
| 解析关键元素（列表页） | `.item.light_odd` 内的 `h2.subtitle a`（角色名）、`a.avatar`（角色链接） |
| 解析关键元素（详情页） | `h1.nameSingle`（角色名）、`.detail`（角色描述文字）、`#infobox`（属性/生日/性别） |
| 图片 | 封面图在 `.infobox .cover` 的 `src`，格式 `//lain.bgm.tv/r/400/pic/crt/...` |

**⑦ 角色评论（char_review）**

| 项目 | 内容 |
|------|------|
| URL | `https://bangumi.tv/character/{charId}` |
| 认证 | 完全公开，无需 cookie（包括 R18 角色） |
| 分页 | **无分页**，所有评论显示在同一页 |
| 解析容器 | `#comment_list`（class 可能为空或有 `commentList comment-list borderNeue`） |
| 解析元素 | `.row_reply` 内的 `.floor-anchor`（楼层/日期）、`strong a.l`（用户名）、`.message.clearit`（文本） |
| 剧透处理 | 内容可能被 `.text_mask > .inner` 包裹，需特殊提取 |
| 已删除内容 | 文本为"删除了回复"，跳过 |
| 注意 | 角色评论和游戏评论是不同的评论区，每个角色独立 |

### 0.3 API 说明

Bangumi v0 API 是稳定的数据获取手段：

- **游戏详情/角色**：`GET /v0/subjects/{id}` / `GET /v0/subjects/{id}/characters` / `GET /v0/characters/{id}`
- **认证**：`Authorization: Bearer <token>`（在 `.env` 配置 `BANGUMI_API_TOKEN`）
- **Token 有效期**：长期有效，可访问所有内容含 R18
- **图片 URL**：直接从 API 返回的 JSON 中获取 `images.medium` / `images.large`

### 0.4 限制与注意事项

| # | 限制 | 当前值 | 影响 | 后续改进方向 |
|---|------|--------|------|-------------|
| 1 | 排行榜页数 `RANK_PAGES = 13` | 取 Top ~300 名 | 仅覆盖排行榜前 300 名，61 页总计可能有 1500+ 名 | 需要时增大页数 |
| 2 | 评分人数门槛 `MIN_RATING_COUNT = 500` | 143 个游戏 | 筛掉了 97 个评论量少的小众游戏（240→143） | 需要时可调低到 100 |
| 3 | 短评只取前 3 页 | ~60 条/游戏 | 热门游戏有上百页评论，**严重不足** | 改成分页爬取或可配置页数 |
| 4 | 长评只取 1 页列表 | 1 页 | 长评可能有多页，**只拿了第一页** | 加分页参数 |
| 5 | 长评全文补爬 | 已补全 1263 篇的正文全文 | 初始只存摘要片段，后通过 `crawl-full-reviews.ts` 补全 | `fullContent` 字段已填充 |
| 6 | 角色详情 `MAX_DETAIL_CHARS = 20` | 前 20 个有完整详情 | 超过 20 个角色的游戏，多余角色无 name_cn/性别/生日 | 增大限制或分批提取 |
| 7 | 角色评论无分页 | 全量一次返回 | 热门角色评论多（如 兰斯 123 条），但仍在合理范围 | 无需改动 |
| 8 | 请求间隔 `REQUEST_DELAY_MS = 1000` | 1 秒 | 143 游戏 × 多 API 调用 = 耗时较长 | 可适当降低到 500ms |
| 9 | 角色评论数 ~2923 个角色 | 2923 次请求 × 1s ≈ 49 分钟 | 角色评论全局一次性爬取 | 已实现断点续爬（按文件存在跳过） |

- **R18 评论/文章/blog**：通过 Puppeteer（Edge 浏览器）已解决。`cookie-refresher.ts` 保持浏览器登录态，`fetchNSFW()` 和 `crawl-full-reviews.ts` 的 `fetchFullReview()` 在普通 `fetch()` 失败时自动走 `browser-fetcher.ts` 通过 CDP 连接真实浏览器获取 HTML。
- **R18 角色评论**：无需特殊处理，角色页公开可访问。
- **API 全量可用**：添加 `BANGUMI_API_TOKEN`（Bearer Token）后，所有游戏详情和角色信息均可通过 API 获取，不再受 R18 限制。
- **公开页面免 Cookie 模式**：`fetchHTML()` 新增 `noCookie` 选项，角色评论等公开页面跳过 cookie 文件读取，减少 I/O 开销。

### 0.7 源码目录职责 — `scripts/lib/`

每个文件只管一件事，互不重叠。跨功能共享工具留在 `lib/` 根，爬虫专用工具归入 `lib/crawl/`。

**`scripts/lib/`（跨功能共享）：**

| `qdrant.ts` | Qdrant REST API 封装：ensureCollection / upsert / upsertBatch / search（支持 filter）/ count / deletePoints |

**`scripts/lib/crawl/`（爬虫专项）：**

**新增依赖**：`cheerio`（HTML 解析）、`dotenv`（环境变量加载）、**`puppeteer`**（Cookie 刷新器，用系统 Edge 浏览器）

**爬虫工作流程：**

```
准备工作 → 运行 cookie-refresher（Puppeteer 打开 Edge，登录后保持会话）
Step 1  → 爬 13 页排行榜 → 拿所有游戏 ID
Step 2  → 筛掉评分人数 ≤ 500 的
Step 3  → 逐个游戏（API 优先 + HTML 评论）：
            ├ API 获取详情（Bearer Token，稳定）
            ├ API 获取角色列表 → 全部角色直拿（含简介/CV），前 20 补性别/生日
            ├ 爬前 3 页短评（实时 cookie 来自 refresher / 浏览器 fallback）
            ├ 爬长评列表（同上）
            └ 存 JSON 到 parsed/{games,comments,reviews,characters}/{id}.json
Step 4  → 打印汇总统计 + 保存失败记录
```

**独立脚本：**
- `crawl-nsfw-comments.ts` — 专门补爬 NSFW 游戏的评论/长评（通过 `fetchNSFW()` 先试普通 fetch，失败则走 `browser-fetcher.ts` 的 Puppeteer CDP 连接）
- `crawl-char-comments.ts` — 遍历所有唯一角色 ID（去重后 ~2923 个），爬取每个角色的吐槽箱评论（无 cookie，无分页，全量一次返回）
- `crawl-full-reviews.ts` — 遍历所有已存 review 条目，逐篇补爬 `/blog/{id}` 获取正文全文存入 `fullContent` 字段（R18 自动走浏览器 fallback）
- `analyze-data.ts` — 分析各数据类型的长度分布，输出分位数统计，指导后续分块策略

**关键设计：**

| 设计 | 说明 |
|------|------|
| 双认证 | API 走 Bearer Token（长期有效）；HTML 评论走实时 cookie（Puppeteer 保持） |

| 角色全量获取 | 角色列表 API 一次返回所有角色（含简介/CV），无需逐个请求 |
| 角色评论无 cookie | 角色吐槽箱完全公开（含 R18 角色），`fetchHTML` 支持 `noCookie` 模式跳过 I/O |

### 1.1 分块策略（数据驱动，依据实际分布定稿）

| 数据源 | metadata.type | 分块策略 | 去噪阈值 | 预估块数 |

| 游戏简介 | `game_intro` | 整段 1 块，不分割 | — | ~140 |
| 角色简介 | `character` | 整段 1 块，不分割 | ≥15 字 | ~3600 |

| 长评 | `review` | 递归分割 `\n\n→\n→。→！？→，`，块 500 字，重叠 50 字 | 仅 2% 噪音，基本全保留 | ~6300 |

### 1.2 批量灌库脚本 — `scripts/etl/process.ts`

将爬取的 parsed JSON 数据**清洗后**批量导入 MongoDB：

流程:
```
parsed JSON 文件 → 加载映射表(gameNames + charNames)
→ 逐类型读取 → 去噪 → 分块（仅长评需要）
→ 打 metadata tag
→ 分批 dedup 查重（每 2000 条一批，防 $in 爆炸）
→ 批量生成 embedding（BGE-m3, 每批 20 条）
→ bulk insert MongoDB
```

#### 分块算法（中文感知递归分割器）

```typescript
function chunkText(text: string, maxSize = 500, overlap = 50): string[]
```

分隔符优先级：`\n\n`（段落）→ `\n`（行）→ `。` → `！？` → `，` → `、` → ` `（空格）→ 硬切

特点：
- 优先在语义边界分割，避免断词
- 若切割点位置 < maxSize × 25%，退化为硬切（防止块太小）
- 硬切时尝试向前找空格降低伤人概率
- 相邻块重叠 50 字（10%），保留上下文

#### 数据清洗规则

把原始数据中不需要的东西去掉，只留下有用的文本：

```
HTML原文: <div class="review-content">这部作品真的很好玩啊<br> 女主角性格很鲜明<br><br> 推荐</div>
清洗后:  这部作品真的很好玩啊\n女主角性格很鲜明\n推荐
```

做法：HTML parser（cheerio）去标签拍平。

| 数据源 | 清洗内容 |
|--------|---------|
| 所有文本 | 纯文本已在 parser 阶段处理，无需重复清洗 |
| 短评 | ≤15 字丢弃；"删除了回复"丢弃 |
| 角色评论 | ≤20 字丢弃；"删除了回复"丢弃 |
| 角色/游戏简介 | 整段保留 |

#### metadata tag 规则

```typescript
{ type: "game_intro",  gameName: "CLANNAD",           source: "game:102" }
{ type: "comment",     gameName: "CLANNAD",           source: "game:102" }
{ type: "review",      gameName: "CLANNAD",           source: "game:102",                     chunkIndex: 0, totalChunks: 5 }
{ type: "character",   gameName: "CLANNAD", charName: "古河渚",          source: "char:123" }
{ type: "char_review", gameName: "CLANNAD", charName: "古河渚",          source: "char:123" }
```

> 注意：`gameName` 通过 `loadGameNames()` 从 `games/*.json` 关联查询。`charName` 通过 `loadCharNames()` 从 `characters/*.json` 关联查询（char_review 必须带 charName，否则自查询时按角色名 filter 会失效）。长评 `review` 的 chunkIndex 从 0 递增，totalChunks 记录总块数。
>
> **设计思路**：metadata 只记"搜索时需要的标签"——用户问"CLANNAD 怎么样" → 搜 `game_intro` + `review`；问"古河渚是个什么样的人" → 搜 `character` + `char_review`。不需要存用户 ID、评分、时间这些 RAG 用不到的字段。

#### 脚本实现演进（V1 → V2 修复记录）

| 问题 | 表现 | 修复 |
|------|------|------|
| `$in` 大数组炸库 | 3 万条 dedupKey 一次性传入 MongoDB，超 16MB BSON 限制 | 分成 **每 2000 条**一批查询 `{ dedupKey: { $in: batchKeys } }`，结果合并到 `existingKeys` Set |
| `char_review` 缺 charName | charName 传空字符串，自查询时 `filter: { charName: "角色名" }` 永远命中不了 | 新增 `loadCharNames()` 从 `characters/*.json` 建立 `charId → nameCN` 映射，灌库时查表填充 |
| 嵌入循环日志语法隐患 | 模板字符串/括号配对易误读 | 拆分为独立数组收集 + 清晰缩进，每 400 条（20×20）打印一次进度 |

#### 幂等性设计

每条记录带 `dedupKey` 字段，格式为 `{type}:{source}:{id}[:chunk:{n}]`。重复运行 `process.ts` 时会跳过已入库的记录，可安全重跑。

#### 字段保留策略（各数据类型对比）

最终存入 MongoDB `resources` 集合的结构：

```ts
{
  content: string,         // 语义搜索的正文
  embedding: number[],     // 1024 维向量
  metadata: {
    type: string,          // 数据类型标签
    source: string,        // 数据来源
    gameName: string,      // 关联游戏名
    charName?: string,     // 角色名 (character/char_review)
    chunkIndex?: number,   // 分块序号 (review)
    totalChunks?: number,  // 总分块数 (review)
  },
  dedupKey: string,        // 幂等去重
}
```

各原始数据类型在灌库时的字段取舍：

| 原始数据类型 | 原始字段数 | **保留到 content** | **保留到 metadata** | **被剔除的字段** |
|---|---|---|---|---|
| **游戏简介** (GameDetail) | 4 | `summary` | `gameId→source`, `nameCN/name→gameName` | 原始 name, nameCN（已折叠进 gameName） |
| **玩家短评** (Comment) | 5 | `text` | `gameId→source`, `gameName` | ❌ userId, userName, score, date |
| **玩家长评** (Review) | 6 | `fullContent` 或 `summary` | `gameId→source`, `gameName`, chunkIndex, totalChunks | ❌ title, author, date |
| **角色介绍** (CharacterInfo) | 5 | `summary` | `charId→source`, `gameName`, charName | ❌ subjectId |
| **角色吐槽** (char-comment) | ~2+ | `text` | `charId→source`, charName | ❌ 其他原始字段 |

一共约 6 个原始字段/类型，保留了 2 个（content + 轻量 metadata），剔除了 4 类：

- 所有**用户身份**（userId、userName、author）— RAG 不关心谁说的
- 所有**分数**（score）— 向量检索不依赖评分
- 所有**时间**（date）— 不按时间排序
- 所有**标题**（title）— 正文已覆盖语义

设计逻辑：RAG 搜索只看语义相似度（content ↔ embedding），metadata 只用来溯源（"哪个游戏/角色的数据"）。用户、评分、时间对向量检索没有意义，占 embedding 上下文长度还稀释语义，直接扔掉。

### 1.5 Embedding 缓存 + 离线写入

#### 背景

`process.ts` 需要同时做两件事：调 embedding API（需代理）和写入 MongoDB（不需代理）。在中国网络环境下，这导致两个问题：

1. **中断风险**：跑 2.8 万条 embedding 生成需要持续调用 SiliconFlow API，一旦网络波动就白跑
2. **重跑成本高**：即使只有 MongoDB 写入失败，重新跑 `process.ts` 又要从头调 embedding API

#### 解决方案：两阶段分离

`process.ts` 在生成所有 embedding 后，先存入本地缓存 JSON，写入 MongoDB 前退出 → 然后由 `load-cache-to-db.ts` 独立完成 MongoDB 写入。

```
process.ts ──→ parsed JSON + chunking
           ──→ generateEmbeddings (需代理, 调 API)
           ──→ 保存 embedded-chunks.json（本地缓存）
           ─────────────────────────── ← 分界线
load-cache-to-db.ts ──→ 读取缓存 JSON（无 API 调用）
                    ──→ dedup 查重
                    ──→ 写入 MongoDB（不需代理）
```

#### 文件

- `scripts/etl/load-cache-to-db.ts` — 独立写入脚本，从本地缓存读取带向量的数据，分批写入 MongoDB
- `scripts/data/cache/embedded-chunks.json` — 本地 embedding 缓存（gitignored）

#### 关键设计

| 设计 | 说明 |
|------|------|
| 两阶段分离 | `process.ts` 只做 embedding（需代理），`load-cache-to-db.ts` 只做写入（不需代理） |
| 幂等写入 | 按 `dedupKey` 分批查重（每 2000 条一批），重复跳过，可安全重跑 |
| 零 API 调用 | 写入阶段不调任何外部 API，纯本地 + MongoDB 操作 |
| 缓存重用 | embedding 缓存保留，下次新增数据时追加写入，已有数据不重复 |

#### 运行方式

```bash
# 先去代理（否则连不上 MongoDB Atlas）
set HTTP_PROXY=
set HTTPS_PROXY=
npx tsx scripts/etl/load-cache-to-db.ts
```

### 1.6 Atlas M0 替代方案探索

#### 痛点

> **注意**：MongoDB Community 版原生不支持 `$vectorSearch`（Atlas / 企业版独占），因此需引入专门的向量数据库 Qdrant。

**架构（实际实现）：**

```
┌─────────────────────────────────────────────────────────┐
│                   腾讯云 4C4G                           │
│                                                         │
│  MongoDB Community（免费）     Qdrant（Docker 容器）     │
│  ┌──────────────────────┐    ┌─────────────────────┐    │
│  │ content              │    │ vector (1024维)      │    │
│  │ metadata             │    │ payload: { content,  │    │
│  │ dedupKey             │    │   metadata, dedupKey }│   │
│  │ embedding (备份)     │    │                      │    │
│  └──────────────────────┘    └─────────────────────┘    │
│           ↑                            ↑                │
│           └──────────┬─────────────────┘                │
│                      │                                  │
│  本地开发 → Qdrant REST API（端口 3933 对外暴露）        │
│            → payload 自带 content + metadata，无需再查   │
│            MongoDB（27017 不对外暴露，仅 server 本地可连）│
└─────────────────────────────────────────────────────────┘
```

> **关键设计**：Qdrant payload 中直接存储了 content + 检索需要的 metadata，搜索返回即可用，无需二次查询 MongoDB。MongoDB 只作为数据备份和离线分析的存储端，不在查询路径上。27017 端口不对外暴露，提高安全性。

| 运维成本 | 零 | 需装 MongoDB + Docker + Qdrant |
| 费用 | 免费 | 服务器已有，零额外费用 |
| 向量搜索 | `$vectorSearch`（内置） | Qdrant REST API |
| 扩展性 | 升 M2+ 付费 | 随服务器资源扩展 |

#### 服务器资源论证：4C4G 3M 跑所有项目够不够

**现有负载：**

| 项目 | 状态 | 资源占用 |
|------|------|---------|
| 静态网页 | 部署中，无访问 | ~50MB |
| 全栈 React + MongoDB | 部署中，无用户 | ~400MB（MongoDB 空闲） |
| C 端 HTML 表单页面 | 待开发 | ~100MB |
| **AI 项目（含向量搜索）** | **即将加入** | 见下文 |

**AI 项目资源占用明细（200K 向量估算）：**

| 资源 | 项 | 估算 |
|------|----|------|
| **内存** | OS 系统 | ~1GB |
| | 现有项目 MongoDB（限 300MB 缓存） | ~300MB |
| | 静态网页 + 全栈 + C 端 | ~400MB |
| | Qdrant（200K 向量本体 + HNSW 索引） | ~1.1GB |
| | Next.js 后端 | ~200MB |
| | **总计** | **~3GB，余 ~1GB** |
| **磁盘** | 80GB SSD，所有项目数据合计 < 5GB | 大量富余 |
| **带宽** | 3Mbps（~384KB/s） | 每次查询传输 < 10KB，响应 < 0.03s |

**关键理解：3M 带宽为什么够用？**

每次 AI 查询的数据流：

```
用户问题（~200 字节）
  → Vercel → Qdrant 搜索（请求 ~4KB，返回 ~200 字节）
           → MongoDB 查文档（请求 ~100 字节，返回 ~5KB）
           → DeepSeek API（等待 1-3s，不经过服务器带宽）
  → 返回给用户（纯文本，几 KB）
```

带宽敏感的大文件（图片/视频/前端资源）走 Vercel CDN，不经服务器。**服务器只处理纯文本的 API 请求，3M 带宽绰绰有余。**

**内存不够怎么办？**

- Qdrant 使用 mmap，热点数据在内存，冷数据自动走磁盘
- 实际运行时 200K 向量只占 ~700MB 常驻内存，不是上述的 1.1GB
- 仍可限流：`Qdrant --memory 1.5G`，MongoDB `--wiredTigerCacheSizeGB 0.3`

**结论：4C4G 3M 服务器跑现有项目 + 200K 向量的 AI 项目，在内存、磁盘、带宽三个维度上都有余量，无需升级配置。**

#### 测试中遇到的问题

| 问题 | 原因 | 解决 |
|------|------|------|
| **convertToModelMessages 运行时崩溃** | AI SDK v6 的 `UIMessage` 使用 `parts` 数组格式（无 `content` 字段），但 `convertToModelMessages` 内部访问 `message.parts.filter()` 时，前端发来的消息是 `{role, content}` 字符串格式，`parts` 为 undefined 导致 `.map` 报错 | 去掉 `convertToModelMessages`，改用自定义 `toSimpleMessages()` 工具函数，从 `m.parts` 中提取 text 拼成 `{role, content}` 格式传给 `streamText` |
| **Qdrant 端口不通** | 腾讯云安全组未放行 3933 端口 | 在安全组控制台手动添加入站规则，放行 TCP 3933 |
| **MongoDB 27017 端口不通** | 安全组未开放，且出于安全考虑不应开放 | 改用 SCP 传输数据文件到服务器，在服务器本地通过 Node.js 脚本写入 localhost:27017 |
| **服务器 mongosh 无 cat()** | MongoDB 8.x mongosh 工具变更 | 改用 Node.js 脚本 + mongodb npm 包进行数据导入 |
| **Qdrant 3933 公网暴露** | 本地开发需要直连服务器 Qdrant，但生产环境不应暴露 | 当前妥协：本地开发期开 3933，等 Next.js 部署到服务器后关闭。届时所有服务在服务器 localhost 内通信 |

```

  你 → 浏览器 → 本地 Next.js(:3000)
```

| 服务 | 当前（本地开发） | 生产目标 |

### 1.9 评估脚本 & Embedding 选型计划

#### 评估脚本目标

Phase 1 收尾的最后一件事——写一个评估脚本，量化对比不同策略的效果。脚本的核心逻辑：

```
手动标 30 条 QA 对
eg. { question: "古河渚是个什么样的人",
       expected: { charName: "古河渚" } }

for each QA pair:
    for each 策略 (纯向量 / +Reranker / +HybridSearch / +QueryRewrite):
        用 question embedding 去 Qdrant 搜 → 看 expected 在前 N 个结果里

输出: Hit Rate@K, MRR
```

这个脚本不只测 chunk 策略，也可以用来对比不同 embedding API：
```
跑一轮用 bge-m3 → 拿到 Hit Rate
再跑一轮用 text-embedding-3-large → 拿到 Hit Rate
对比哪个 embedding 在你自己的数据上效果更好
```

**P0 优先级**：与爬虫平级。没有评估数据，面试只能说"我感觉变好了"；有了评估数据，能直接说"Reranker 提升了 14 个点的命中率"。

#### 基线评估结果（2026-05-17）

运行 `pnpm tsx scripts/eval/evaluate.ts` 对 30 条 QA 对（10 事实类 + 20 观点类）的测试结果：

| 分类 | 匹配方式 | Hit@1 | Hit@3 | Hit@5 | MRR | 命中/总数 |
|------|---------|-------|-------|-------|-----|---------|
| **全部 (30 题)** | 严格 | 53.3% | 63.3% | 63.3% | 0.578 | 19/30 |
| | 模糊 | 73.3% | 90.0% | 90.0% | 0.811 | 27/30 |
| **事实类 (10 题)** | 严格 | 80.0% | 90.0% | 90.0% | 0.850 | 9/10 |
| | 模糊 | 90.0% | **100.0%** | **100.0%** | 0.950 | 10/10 |
| **观点类 (20 题)** | 严格 | 40.0% | 50.0% | 50.0% | 0.442 | 10/20 |
| | 模糊 | 65.0% | **85.0%** | **85.0%** | 0.742 | 17/20 |

##### 对评测脚本的质疑与修正

第一版 QA 对全是"事实查表"型问题（"古河渚是什么样的人" → 按 charName 搜角色简介），本质上就是用向量搜索查索引里已有的名字，必然命中。**这测试的是搜索引擎，不是 RAG。**

RAG 真正的价值是从碎片评论中召回**相关观点**，而不是查简介。因此将 QA 对重构为两类：

| 类型 | 特点 | 示例 | 预期 |
|------|------|------|------|
| **事实类 (fact)** | 角色简介/游戏介绍，查表即可 | "古河渚是什么样的人"、"命运石之门是什么样的游戏" | ≈100%，底线验证 |
| **观点类 (opinion)** | 评论/评价/对比，需语义匹配 | "CLANNAD 哪个情节最催泪"、"素晴日的哲学是不是掉书袋" | RAG 核心指标 |

结果符合预期：事实类模糊匹配 100%，观点类 85%（纯向量搜索）。后续加 Reranker/Hybrid Search 后，观点类的提升才是真正的价值。

##### 全 Phase 可重跑设计

评估脚本的 `STRATEGIES` 数组支持注册多个策略函数，每个 Phase 加完新功能后注册一个新策略：

```typescript
const strategies = [
  { name: '纯向量搜索', fn: baselineStrategy },       // Phase 1 baseline
  // { name: '+Self-Querying',  fn: selfQueryStrategy },  // Phase 2
  // { name: '+Reranker',       fn: rerankerStrategy },   // Phase 3
  // { name: '+HybridSearch',   fn: hybridStrategy },     // Phase 4
  // { name: '+QueryRewrite',   fn: rewriteStrategy },    // Phase 3
]
```

跑一次出所有策略的对比表，`evaluation-result.json` 也按策略分别存指标。即使所有 Phase 都跑完、数据更新到 200K，也只需执行 `pnpm tsx scripts/eval/evaluate.ts` 一条命令，所有策略重新跑一轮，diff 出对比。

##### 数据命名问题

差距主要来自**数据命名不一致**：

| 问题类型 | 例子 | 影响 |
|---------|------|------|
| 游戏名带子标题 | "白色相簿2" 存为 "白色相簿2 序章/终章" | 严格匹配不到 |
| 角色名存日文原名 | "冬马和纱" 存为 "冬馬かずさ" | 严格匹配不到 |
| 游戏名存中文全称 | "素晴日" 存为 "美好的每一天 ～不连续的存在～" | 严格/模糊均不到 |
| 数据本身未收录 | 秽翼的尤斯蒂娅的 charName 未收录中文名 | 无法命中 |

这暴露的**不是检索策略的问题，是数据清洗的问题**。修复方向：灌库时给 gameName/charName 额外存一个 `nameCN` 别名。注意需要确保别名的一致性——翻译过来的中文名不一定是游戏的实际常用名（例如 Bangumi 存的是"美好的每一天"但玩家叫"素晴日"）。具体修复推迟到 Phase 2 之后。

##### 评估增强（2026-05-17 第二版）

第一版 30 条 QA 对 + 仅 Hit Rate/MRR 两个指标，对于 200K 知识库覆盖面不够。第二版从三个维度增强：

**① QAPair 接口增强**

```typescript
interface QAPair {
  id: string
  category: 'fact' | 'opinion' | 'comparison'
  question: string
  expect: Record<string, string>     // 主要匹配条件
  theme: string                      // 新增：主题（游戏名/角色名），用于 Precision@K
  expectType?: string                // 新增：期望的数据类型
  note: string
}
```

**② 新增指标**

| 指标 | 计算方式 |
|------|---------|
| **Precision@K** | top-K 中 payload 匹配 theme 的比例 |
| **Type Accuracy** | top-K 中 type 匹配 expectType 的比例（仅已标注的 QA） |
| **Source Diversity** | top-K 中 unique 的 gameName 数量 |

按 all / fact / opinion / comparison 四组分别输出。

**③ QA 对扩展：30 → 80+**

新增 ~50 条 QA 对，全部基于 Qdrant 真实数据（142 个游戏名、1475 个中文角色名、5 种数据类型）。覆盖：
- 现有事实类 QA 补充 theme 字段
- 新增观点类跨游戏/角色评论检索
- 新增 comparison 类跨游戏对比（limit 提升到 20）
- 覆盖短查询、泛查询、别名查询等边界场景

**④ 评估原则**

## Phase 2：自查询检索（Self-Querying）

### 动机

当前检索无 filter，TypeAcc ~0.4%（混杂 5 种 type），对比类 AnsRelevance 82.8%（context 常缺一边）。Self-Querying 通过 Qdrant filter 按 type/gameName/charName 预过滤候选集，预期：
- TypeAcc 0.4% → **80%+**
- 对比类检索命中显著改善（`should` 多路召回）
- 事实类 HitRate@1 提升（直过滤到目标游戏）
- 生成阶段 context 纯度提高 → AnsRelevance 同步提升

### 前置条件：Qdrant Payload 字段

Qdrant `resources` 集合中每条记录有以下可过滤字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `type` | string | 数据类型：`game_intro` / `comment` / `review` / `character` / `char_review` |
| `gameName` | string | 游戏名称（含 nameCN 和别名） |
| `charName` | string | 角色名称 |
| `source` | string | 数据来源标识 |

Qdrant filter 支持 `match`（精确匹配）和 `range`，无需预建索引。

### 2.1 更新 `src/lib/qdrant.ts`（searchSimilar 加 filter）

```typescript
// 签名增加可选 filter
export async function searchSimilar(
  vector: number[],
  limit = 3,
  filter?: { must?: QdrantCondition[]; should?: QdrantCondition[] },
): Promise<QdrantSearchResult[]>

// filter 存在时，在 Qdrant 请求 body 加入 filter 字段
// 单字段精确匹配：
filter = { must: [{ key: "type", match: { value: "game_intro" } }] }
// 多字段 AND：
filter = { must: [{ key: "type", match: { value: "game_intro" } }, { key: "gameName", match: { value: "CLANNAD" } }] }
// OR 匹配（对比类用）：
filter = { should: [{ key: "gameName", match: { value: "白色相簿2 终章" } }, { key: "gameName", match: { value: "CLANNAD" } }] }
```

### 2.5 第一次优化：别名展开 + TypeAcc 修正

#### 2.5.1 优化方案

**问题**：V1 filter 用 `must`（精确 AND），导致：
- `gameName: "白色相簿2"` 不匹配 payload `"白色相簿2 终章"`
- `TypeAcc` 因 `top5.length >= 5` 检查在 filter 返回不足时恒为 0

**改动**（scripts/eval/evaluate.ts）：

1. **别名展开**：`must` → `should + min_should: 1`
   - 单 gameName：`should: [gameName, gameAliases], min_should: 1`
   - 单 charName：`should: [charName, charNameCN, charAliases], min_should: 1`
   - 对比类：每个 gameName 展开 `[gameName, gameAliases]`
   - type 过滤保留 `must`（与别名 OR 组合）

2. **TypeAcc 公式修正**：
   - 改前：`top5.length >= 5 ? ... / 5 : 0`

## Phase 3：查询重写 + 重排序

### 3.1 Reranker — `src/lib/ai/reranker.ts`（新文件）

调 SiliconFlow rerank API，`$vectorSearch` **Top 20** → rerank → **Top 5**：

```typescript
POST https://api.siliconflow.cn/v1/rerank
Authorization: Bearer <SILICONFLOW_API_KEY>
```

> 候选文档数从 10 提到 20，因为 rerank 处理 20 条的开销很小，能多召回 10 条可能漏掉的边缘相关文档，然后精排。

### 3.2 rewriteQuery Tool（新）

独立 tool，AI 在搜索前先调用，将口语问题改写为检索友好的关键词组合：

- 输入：用户原始问题
- 输出：改写后的搜索查询（去口语化、提取关键实体）
- 实现：调 DeepSeek 模型做一句话改写
- 适用场景：
  - "那个白毛红瞳的女生是谁" → "白色头发 红色眼睛 女性 角色"
  - "这游戏剧情怎么样" → "剧情 评价"
  - 日语罗马音 → 中文（如果数据库里存的是中文名）

- RRF（Reciprocal Rank Fusion）合并结果：

  ```typescript
  score = 1 / (60 + rank_vector) + 1 / (60 + rank_text)
  ```

- **注意**：MongoDB `$text` 对中文支持有限（不分词，按字索引），实测效果可能不如纯向量。若差异大则降级回纯向量搜索，需要中文分词时考虑 Elasticsearch（超出当前项目范围）。

### 4.2 前端体验增强（后续，优先级最低）

- 引用溯源：AI 回复中的观点可点击定位到原始评论
- Token / 成本可视化
- Tool Calling 增强：搜索实时评分 + 本地 RAG 结合

| 选型 | 论证角度 |
