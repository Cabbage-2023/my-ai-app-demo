## 📅 2026-05-15 阶段：突破 Bangumi 会话反爬机制

- **踩坑记录**：Axios 请求频繁遭遇 502/403，发现核心 Session ID `chii_sid` 被服务器强制高频刷新（约 1 分钟过期），导致传统静态 Cookie 爬虫彻底瘫痪。
- **攻坚方案**：
  1. 放弃朴素的 HTTP Client 硬怼，引入 Puppeteer 托管真实 Chromium 浏览器。
  2. 实现了"注入初始 Cookie → 模拟人类浏览行为随机延迟（3-5s）→ 浏览器自动维持和更新 Session"的自动化方案。
  3. 通过配置 `.npmrc` 切换至阿里云国内镜像，解决了 Chromium 150MB 底层二进制包在国内环境下的网络下载卡死问题。

## 📅 2026-05-16 阶段：数据画像与全量清洗 + 分块策略定稿

- **当时现状**：全量抓取了约 50,000 条多维数据（短评、长评、角色等）。长评发现只有 128 字摘要，属于严重残缺数据。
- **神级数字（核心吹点）**：跑了 `analyze-data.ts` 拿到关键画像：
  - 角色评论：中位数 11 字，≤15 字噪音占比高达 **64%**
  - 短评：≤15 字噪音占比 **29%**
  - 长评：78% 大于 500 字，中位数 1288 字，最长超 8 万字
- **硬核决策**：
  1. 绝不留技术债，连夜编写 `patch-reviews.ts` 补全 1263 篇长评全文，打通 Parent-Child 分块链路。
  2. **差异化阈值去噪**：短评去噪阈值设为 15 字（剔除 29% 垃圾）；角色评论果断提至 20 字，定向爆破 64% 低语义情绪水评。
  3. **保持原子独立性**：短评与角色评论坚持"单条成块，绝不拼凑"，规避多用户主观视角强行交织带来的多主体交叉污染。
  4. **长文本标点降级**：对长评全文定制 500 窗口 + 50 重叠度，启用 `\n\n → \n → 。 → ，` 的强语义标点递归切分器。
- **阶段成果**：全站数据从 50,000 条粗糙文本，精炼为约 **29,000 个高质量 Chunk**。帮 Atlas M0 省了约 60% 存储成本，同时净化了向量空间。

## 📅 2026-05-17

### 阶段 1：Atlas $vectorSearch → 腾讯云自托管 Qdrant 迁移

- **迁移原因**：Atlas M0 免费层在大批量写入时 IOPS 严重受限（灌 3 万条需数小时），且 MongoDB Community 版不支持 `$vectorSearch`（Atlas/企业版独占）。需要在不增加额外费用的前提下，把向量搜索从 Atlas 迁移到自建基础设施。

- **架构决策**：引入 Qdrant 作为专用向量数据库。初始设计是 "Qdrant 搜向量 → 拿 mongoId → MongoDB 查原文"的两段式，但在实现中发现 Qdrant payload 可以直接存储 content + metadata，搜索返回即可用，无需二次查询 MongoDB。最终架构简化为 **Qdrant 负责向量搜索 + 数据返回，MongoDB 仅做备份存储**。

- **安全组配置**：腾讯云安全组开放 Qdrant 端口 3933（REST API），MongoDB 端口 27017 保持关闭。本地开发（Next.js localhost:3000）通过公网 Qdrant REST API 查询，MongoDB 数据通过 SCP 传输到服务器后本地写入。

- **双写一致性**：`addResource` tool 新增数据时同时写入 MongoDB（`Resource.create()`）和 Qdrant（`upsertPoint()`）。灌库脚本 `scripts/load-to-qdrant.ts` 从本地 embedding 缓存读取，分批写入 Qdrant（每批 100）+ MongoDB（每批 500，带 dedup）。

- **数据迁移结果**：SCP 传输 367MB 的 `embedded-chunks.json`（28602 条带向量数据）到服务器。服务器本地 Node.js 脚本写入 Qdrant 28602 条向量，MongoDB ai_rag 数据库写入 27444 条文档（去重跳过 3757 条）。

- **MongoDB 复用策略**：服务器已有 wenjuan-platform 项目的 wenjuan-mongo Docker 容器，在此容器内新建 `ai_rag` 数据库。物理上同一 MongoDB 进程（减少内存开销），逻辑上数据库隔离。无需再起一个 MongoDB 容器。

- **TypeScript 兼容问题**：AI SDK v6 的 `convertToModelMessages` 函数要求消息含 `parts` 数组字段（新的 UIMessage 格式），但前端发送的是简单 `{role, content}` 字符串格式，导致运行时崩溃。最终去掉 `convertToModelMessages`，改由自定义 `toSimpleMessages()` 从 `m.parts` 提取 text 组装 `{role, content}`。`streamText` 天然兼容两种格式，无需转换层。

### 阶段 2：RAG 检索质量评估脚本

- **动机**：没有评估，做了六个功能但说不清楚每个带来了什么提升。面试时只能讲"我感觉变好了"，不能甩数据。

- **评估设计**：30 条 QA 对，手动标注期望文档的 payload 条件（如 `{ charName: "古河渚" }`），每条问题去 Qdrant 搜索，看期望文档在 Top-K 中的位置。输出 Hit Rate@K 和 MRR。

- **关键质疑与重构**：第一版 QA 对全是"事实查表"型——"古河渚是什么样的人"按 charName 搜角色简介，本质上就是在测向量搜索能不能找到同名文档，必然命中。**这测的是搜索引擎，不是 RAG。** RAG 真正的价值是从碎片评论中召回相关观点。重构为两类：

  | 类型 | 数量 | 说明 |
  |------|------|------|
  | 事实类 (fact) | 10 题 | 角色/游戏简介查表，≈100% 预期 |
  | 观点类 (opinion) | 20 题 | 评论/评价语义匹配，RAG 核心指标 |

- **基线结果（纯向量搜索）**：

  | 分类 | Hit@1 (模糊) | Hit@3 (模糊) | MRR |
  |------|:---:|:---:|:---:|
  | 全部 30 题 | 73.3% | 90.0% | 0.811 |
  | 事实类 10 题 | 90.0% | **100%** | 0.950 |
  | 观点类 20 题 | 65.0% | **85.0%** | 0.742 |

  事实类接近 100% → 向量搜索基本功没问题。观点类 85% Hit@3 → 纯向量基线还可以，后续加 Reranker/Hybrid Search 后观点类的提升才是真正的价值。

- **全 Phase 可重跑设计**：`STRATEGIES` 数组模式，每个 Phase 加完功能后注册一个新策略，一条 `pnpm tsx scripts/evaluate.ts` 跑所有策略对比。

- **第二版增强（82 条 QA，2026-05-17）**：QA 对从 30 扩展到 82（15 事实 + 51 观点 + 16 对比），全部基于 Qdrant 真实数据。新增 Precision@K、Type Accuracy、Source Diversity 指标。

  **第二版命中率：**

  | 分类 | Hit@1 (模) | Hit@3 (模) | MRR |
  |------|:---:|:---:|:---:|
  | 全部 82 题 | 70.7% | 82.9% | 0.789 |
  | 事实类 15 题 | 80.0% | **100%** | 0.889 |
  | 观点类 51 题 | 70.6% | 78.4% | 0.770 |
  | 对比类 16 题 | 62.5% | 81.3% | 0.757 |

  **第二版质量指标（模糊匹配）：**

  | 分类 | Prec@5 | TypeAcc | 多样 |
  |------|:---:|:---:|:---:|
  | 全部 | 55.4% | 0.5% | 2.15 |
  | 事实类 | 68.0% | 2.7% | 1.67 |
  | 观点类 | 56.1% | 0.0% | 2.22 |
  | 对比类 | 41.3% | 0.0% | 2.38 |

  **关键发现**：
  - Type Accuracy 仅 0.5% → 搜游戏简介时 top-5 大多是评论而非 `game_intro`，Phase 2 type filter 预期从 2.7% → 90%+
  - 对比类严格 Hit@3=43.8%，模糊 81.3% → 语义相关但命名不一致
  - 4 题完全未命中：Saber（英文无数据）、素晴日掉书袋（语义鸿沟）、评价音乐画面（无锚点）、柚子社推荐（别名不匹配）
  - Prec@5≈55% → top-5 约一半与主题相关，Reranker（Phase 3）目标是提到 80%+
