## 📅 2026-05-15 阶段：突破 Bangumi 会话反爬机制

- **踩坑记录**：Axios 请求频繁遭遇 502/403，发现核心 Session ID `chii_sid` 被服务器强制高频刷新（约 1 分钟过期），导致传统静态 Cookie 爬虫彻底瘫痪。
- **攻坚方案**：
  1. 放弃朴素的 HTTP Client 硬怼，引入 Puppeteer 托管真实 Chromium 浏览器。
  2. 实现了"注入初始 Cookie → 模拟人类浏览行为随机延迟（3-5s）→ 浏览器自动维持和更新 Session"的自动化方案。
  3. 通过配置 `.npmrc` 切换至阿里云国内镜像，解决了 Chromium 150MB 底层二进制包在国内环境下的网络下载卡死问题。

## 📅 2026-05-16 阶段：数据画像与全量清洗 + 分块策略定稿

- **当时现状**：全量抓取了约 50,000 条多维数据（短评、长评、角色等）。长评发现只有 128 字摘要，属于严重残缺数据。
- **神级数字（核心吹点）**：跑了 `analyze-data.ts` 拿到关键画像：
  - 角色评论：中位数 11 字，≤15 字噪音占比高达 **64%**
  - 短评：≤15 字噪音占比 **29%**
  - 长评：78% 大于 500 字，中位数 1288 字，最长超 8 万字
- **硬核决策**：
  1. 绝不留技术债，连夜编写 `patch-reviews.ts` 补全 1263 篇长评全文，打通 Parent-Child 分块链路。
  2. **差异化阈值去噪**：短评去噪阈值设为 15 字（剔除 29% 垃圾）；角色评论果断提至 20 字，定向爆破 64% 低语义情绪水评。
  3. **保持原子独立性**：短评与角色评论坚持"单条成块，绝不拼凑"，规避多用户主观视角强行交织带来的多主体交叉污染。
  4. **长文本标点降级**：对长评全文定制 500 窗口 + 50 重叠度，启用 `\n\n → \n → 。 → ，` 的强语义标点递归切分器。
- **阶段成果**：全站数据从 50,000 条粗糙文本，精炼为约 **29,000 个高质量 Chunk**。帮 Atlas M0 省了约 60% 存储成本，同时净化了向量空间。

## 📅 2026-05-17 ~ 5-18 凌晨：Phase 1 收尾

### 阶段 1：Atlas → 腾讯云自托管 Qdrant 迁移

- **动因**：Atlas M0 免费层写入 IOPS 瓶颈（3 万条数小时），且 MongoDB Community 不支持 `$vectorSearch`。
- **方案**：服务器已有 wenjuan-mongo Docker 容器内新建 `ai_rag` 库 + Qdrant 容器（端口 3933 开放，27017 关闭）。
- **架构简化**：Qdrant payload 直接存 content+metadata，搜索返回即用，无需二次查 MongoDB。MongoDB 仅做备份。
- **结果**：SCP 传输 367MB 数据，Qdrant 写入 28,602 条向量，MongoDB 写入 27,444 条文档（去重 3,757）。

### 阶段 2：评估脚本迭代（30 → 82 → 150 QA 对）

- **V1（30 QA）**：事实类模糊 100%，观点类模糊 85%。发现第一版 QA 纯查表→重构为 fact/opinion/comparison 三类。
- **V2（82 QA）**：新增 Precision@K、TypeAcc、Diversity 指标。事实类 100%，观点类模糊 Hit@3=78.4%，对比类 81.3%。
- **V3（150 QA）**：修复对比类匹配逻辑（要求所有 expect 同时出现），对比类模糊 Hit@3=60.0%（回归真实）。Prec@5=60.3%。

### 阶段 3：NameCN 别名系统（1.11）

- 手工维护别名映射（~20 游戏短名 + ~12 品牌 + ~10 角色中文名），在 `load-to-qdrant.ts` 中用 `computeNameAliases()` 写入 payload 字段 `gameAliases`/`charNameCN`/`producer`。

### 阶段 4：LLM 辅助别名 + 补爬（1.12 ~ 1.13）

- `scripts/utils/suggest-aliases.ts` 调 LLM 为游戏中角色推荐常用译名/俗称，输出 review 文件手工确认。
- 发现 rank-page-2/5/7 爬虫缓存为 502 错误页（`fetchHTML` 未检查 `res.ok`），导致 ~70 游戏永久缺失。
- 补爬 51 游戏 + 增量 embedding，Qdrant 升至 34,984 点。Merge 别名：114 游戏别名 key + 770 角色别名 key。
- `computeNameAliases()` 新增 `charAliases` 字段，evaluate.ts 同步支持。

### 阶段 5：全量短评+长评补爬 & JSONL 迁移（1.14，5-18 凌晨）

- **动因**：短评只爬了前 3 页（热门游戏有几十上百页），长评只有第 1 页无分页。补爬后缓存膨胀至 633MB，`JSON.stringify` 触发 `RangeError: Invalid string length`。
- **方案**：`crawl-more-pages.ts` 遍历 194 游戏补爬短评第 4~20 页 + 长评第 1~10 页；缓存格式全面从单文件 JSON 切换为**流式 JSONL**（逐行读写，不限文件大小）。

- **结果**：Qdrant 从 34,984 → **49,367** 点（+40.6%），新增 23,941 短评 + 1,050 长评。
- **评估**：数据量增长 40%，模糊 Hit@5 维持 95.3% 未降质；对比类模糊 Hit@5 +4pp（84→88%）。详细指标见项目计划 1.14。

## 📅 2026-05-19：Phase 3 检索质量提升 & Phase 4 RAGAS + 后端韧性

### Phase 3 — Reranker + Query Rewrite + 别名映射 + 置信度阈值

**Reranker 接入 SiliconFlow**：
- 方案：Qdrant 粗召回 Top 20（向量余弦 0.69-0.73）→ rerank 精排 → Top 5（语义相关 0.988-0.994）。区分度从[0.69,0.73]拉到[0.988,0.994]，边界清晰。
- 降级策略：API key 未配 / HTTP 非 2xx / 网络异常 → 分别打印错误后返回原始 topN。核心链路不依赖 reranker 可用性。
- **决策**：候选数从 10 提至 20。理由：rerank 20 条额外开销极小（ms 级），能多捞 10 条边缘相关文档。

**Query Rewrite — Tool 而非自动节点**：
- 踩坑警示：`nodes.ts` 中原有的 `rewriteNode`（纯函数节点）从未接入 graph，属于死代码。本次新增 `rewriteQueryTool` 作为显式 Tool 供 AI 自主调用。
- **决策**：AI 自主判断是否改写 vs 系统自动改写。选了 Tool 方案——简单问题（"CLANNAD 评分多少"）改写不改写效果一样，Tool 方式能省一次调用。实测四类改写均成功（口语→关键词、模糊→具体、罗马音→中文、口语省略→标准）。

**别名映射 — Qdrant min_should 格式踩坑**：
- Qdrant `min_should` 不是 Filter 顶层属性。正确格式：`must: [{ min_should: { conditions: [...], min_count: 1 } }]`。直接写 `{ min_should: ... }` 会静默忽略。
- 别名展开逻辑：`gameName` + `gameAliases`（数组）用 min_should 并联，任一匹配即命中。角色同理多查 `charNameCN` + `charAliases`。
- **效果**：过滤器不再放过别名数据，直接解决 Phase 2 尾部的 ② 别名漏召回。

**置信度阈值 — 0.4/0.7 三层决策**：
- >0.7：正常返回 Top 5。0.4-0.7：metadata 附带分数，LLM 自行判断置信度。<0.4：返回空数组，LLM 自然 fallback。
- **决策**：不写死"置信度低时追加澄清文案"规则。relevance_score 已随结果透传，LLM 感知到分数后自然会追加澄清或转搜索。后续如需细化在 system prompt 补指令即可。

### Phase 4 — RAGAS + 后端中止 + 前端修复

**RAGAS 版本地狱（0.4.x vs 0.2.x）**：
- 最新 RAGAS 0.4.x API 全线变更：`LangchainLLMWrapper` 移除→`llm_factory()`，metrics 从模块级单例改为构造函数传参（`Faithfulness(llm=llm)`），import 路径全改。
- **锁定 ragas==0.2.15**。0.2.15 用旧 API 正常跑完全部指标。
- **Windows 编码踩坑**：子进程 stdin 传中文 JSON 时，Windows 终端（chcp 936 GBK）损坏 Unicode。方案：`PYTHONIOENCODING=utf-8` + `sys.stdin.reconfigure(encoding='utf-8')`。
- **0.2.15 采坑**：`result['faithfulness']` 列访问抛 `KeyError`，需 `result.to_pandas()` 兜底取值。

**RAGAS 评估结果（150 QA × 3 指标 = 450 次调用）**：
- **Faithfulness 84.0%** — 比自写 judge（97.1%）严格 ~13pp。RAGAS 拆原子声明逐条核对，市面上很多 RAG 系统在 70-80%。中位数 90.9% 说明大部分回答干净，低分集中在少数问题。
- **Context Recall 62.6%** — 中位数 100%，说明"找得到就全找到，找不到就 0"。56 题低于 50% 的问题集中在冷门角色和跨系列对比。**根因是知识库覆盖盲区，非搜索链路质量**。
- **Answer Relevancy 50.4%** — bge-m3 余弦相似度对措辞和长度高度敏感。当前 reference 每条 30-80 字，LLM 回答是完整复述，余弦距离天然偏大。反映的是 embedding 特性而非回答质量。
- **关于 reference 的随意性**：观点类问题不存在标准答案。100 题观点类的 context_recall 意义有限——low recall 可能是 reference 只是评价的一家之言。后续：事实类保留context_recall，观点类仅依赖 faithfulness + answer_relevancy。

**Backend Abort — req.signal 不可靠的教训**：
- 前端 `useChat().stop()` abort fetch 后，`req.signal` 在服务端不一定触发——浏览器可能 keep-alive 而非关闭 TCP。
- 加 `ReadableStream.cancel()` 双重保障。但 `createUIMessageStreamResponse` 内部 `pipeThrough` 链不一定传播 cancel 到源头。
- 最终验证：两种信号都能触发。但旧 catch 块在 `AbortError` 后还尝试 `controller.enqueue()` 发错误消息给用户，流已取消时 enqueue 失败导致信号丢失。
- **修复**：catch 块识别 `AbortError` → 发送干净 `finishReason: 'stop'` 收尾，不再报错。经测试：点 stop 后 `req.signal` + `cancel()` 双重触发，当前 `model.invoke()` 抛出 `AbortError`，干净收尾，无多余 token。

**前端 Hydration + 级联渲染踩坑（React 19）**：
- **坑 1**：`useEffect` 内同步 `setState` 读 localStorage → React 19 警告"cascading renders"（浏览器布局完成后强制第二帧）。
- **坑 2**：改 `useState(() => localStorage.getItem(...))` lazy initializer 想跳过 effect → 服务端 SSR 返回 false，客户端 hydration 读到 `"dark"` 返回 true → **Hydration failed**，整棵树抛弃重来。
- **根因**：lazy initializer 在 SSR 调和阶段同步执行，比 `useEffect` 更早。React 要求首次客户端渲染与 SSR HTML 完全一致，任何 browser-only API 分支都会触发 hydration error。
- **最终方案**：`useSyncExternalStore` 从 `<html>` class 派生状态，MutationObserver 订阅变更，`useEffect` 仅操作 DOM（无 setState）。消除了级联渲染 + hydration 错误 + setState 警告三合一。

**知识库卡片闪烁修复**：
- 老逻辑用 `allCompleted` 等所有工具完成才渲染卡片。Agent 多轮调 tool（先查 Key 社作品→再查每部游戏详情）追加新 parts 后 `allCompleted` 变 false → 卡片消失 → 新批次完成 → 重新渲染。

## 📅 2026-05-20：Phase 4 混合搜索 + RAGAS + 前端增强 & Phase 5 RAG Agent 能力升级

### Phase 4.1 — 混合搜索（Hybrid Search）

**动因**：dense-only 搜索搜不到短名/缩写（如"LB"在语义空间中模糊），Qdrant 原生稀疏向量无需新增中间件即可双路检索。

**方案**：
- 轻量级 token-hash 稀疏向量生成器（`src/lib/ai/sparse-embedding.ts`），无外部依赖、无需模型下载
- 同一 Qdrant collection 存 dense（1024d BGE-M3）+ sparse（BM25 风格）双向量
- 检索改为 dense + sparse 双路 → RRF 融合 → reranker → 置信度 0.4 阈值
- 存量 ~50K 数据通过 `backfill-sparse.ts` 补 sparse vector

**效果**：短名/缩写不再依赖别名表兜底，评论正文中出现的"LB""CL"等被 sparse vector 自动命中。RRF 互补 dense 语义 + sparse 词面匹配。

**踩坑**：
- SiliconFlow BGE-M3 API 不支持 sparse 输出，只能自实现 token-hash
- Qdrant sparse_vectors 需在创建 collection 时指定，无法 PATCH，存量需重建 collection

### Phase 4.2 — RAGAS 评估集成

**方案**：Python 子进程桥接，`scripts/eval/RAGAS/` 目录，锁定 `ragas==0.2.15`。

**RAGAS 结果（150 QA × 3 指标）**：
- **Faithfulness 82.5%** — 比自写 judge（~98%）严格 ~16pp，中位数 90.2% 说明大部分回答干净
- **Context Recall 53.7%** — 中位数 90%，对比类 reranker 后下降（86%→69.3%），根因是单一评价性数据源
- **Answer Relevancy 50.6%** — bge-m3 余弦相似度对措辞长度敏感，embedding 特性而非回答质量问题

**混合搜索 vs dense-only 对比**：Faithfulness -1.5pp / Answer Relevancy +0.2pp / Context Recall -8.9pp，均在噪声范围。混合搜索的价值更多体现在实际使用中的短名命中，非 RAGAS 指标可捕捉。

**踩坑**：
- RAGAS 0.4.x API 全线变更（`LangchainLLMWrapper` 移除、metrics 构造函数传参），锁定 0.2.15
- Windows 中文编码：`PYTHONIOENCODING=utf-8` + `sys.stdin.reconfigure(encoding='utf-8')`

### Phase 5 — RAG Agent 能力升级

**5.0 记忆体系**：三种记忆污染防控设计 —— Prompt 记忆（不入库）、对话 RAG 记忆（独立 collection）、主库补充（严格质量门禁）。砍掉 `addResource` tool。

**5.2 多轮上下文**：12 条消息阈值 → DeepSeek 压缩最早 N 轮为摘要，通过 `state.context` 动态注入 system prompt。首版全量重压缩。

**5.3 多步推理 + 主动澄清**：
- 批量查询模式：`getInformation` 支持 `questions` 数组参数，`executeSearch()` 独立函数，`Promise.all` 并行 + 去重
- 动态步数控制：system prompt `## 步数控制` 自评指令
- 主动澄清：`disambiguateEntity` tool，模糊指代时反问用户确认
- 5.3 → 5.6 桥接：质量门禁（type=4、去重检查、简介长度）→ `/api/backfill`

**5.4 长期会话记忆**：`conversation_memory` collection，对话结束时 LLM 总结 → embedding → Qdrant。三重触发（beforeunload + sendBeacon / 30min idle / 新建对话），30s 去重。新对话按 query 检索注入 system prompt。

**5.5 缓存学习**：按计划低优先级推迟。sparse vector 已覆盖短名匹配，别名表退守 metadata 映射场景。
